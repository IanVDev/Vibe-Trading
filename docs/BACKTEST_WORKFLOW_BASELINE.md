## 1. Objective: deterministic pre-LLM workflow for simple MA-crossover backtest

The agent must answer canonical MA-crossover backtest prompts with
deterministic numeric metrics (Sharpe ratio, max drawdown, total return,
number of trades) without depending on the LLM to orchestrate
`config.json → signal_engine.py → backtest → metrics.csv → interpret`.
Local LLMs were empirically unable to compose that orchestration reliably;
Patch 7 replaces it with code.

## 2. Simple MA-crossover backtest prompts are intercepted before the LLM/ReAct loop

`AgentLoop.run()` invokes `BacktestWorkflowDispatcher().try_route(...)`
after the `MarketDataDispatcher` and `CandlestickWorkflowDispatcher`
short-circuits and before the ReAct `while` loop. If the dispatcher returns
a result dict, the loop short-circuits and the LLM is never called. If it
returns `None`, the ReAct loop runs unchanged.

## 3. `backtest_workflow` is the deterministic entrypoint

Defined in `agent/src/agent/backtest_workflow.py` as
`BacktestWorkflowDispatcher.ROUTED_BY = "backtest_workflow"`. The
dispatcher emits trace events tagged with this name so any auditing tool
can filter on it.

## 4. `signal_engine.py` is generated from a fixed template, never by the LLM

The dispatcher renders `code/signal_engine.py` by calling
`_MA_CROSSOVER_TEMPLATE.format(fast_window=..., slow_window=...)`.
The template is a string constant in `backtest_workflow.py` — it is never
fetched from a URL, never generated by an LLM call, and never passed
through `eval()` or `exec()`. Any deviation from this pattern is a
baseline regression.

## 5. The LLM does not decide the first action for simple MA-crossover backtest

Confirmed by the integration test
`test_loop_does_not_call_llm_for_candlestick_prompt` (Level 2) and the
equivalent backtest routing tests. The short-circuit result carries
`iterations == 0` and `routed_by == "backtest_workflow"`.

## 6. Dispatch order: `write_config` → `write_signal_engine` → `backtest` → `read_metrics` → `answer`

The dispatcher executes these five steps in strict order. The trace records
each step with a `workflow_step` event so any inversion or skip is
immediately visible. Any deviation is a baseline regression.

## 7. Required metrics: Sharpe ratio, max drawdown, total return, number of trades

The result dict must expose `sharpe`, `max_drawdown`, `total_return`, and
`trade_count`. Each value must be parseable as a number. Missing or
non-numeric values trigger a fail-closed error, not a fabricated answer.

## 8. Metrics must come from `artifacts/metrics.csv`

The dispatcher reads metrics from `<run_dir>/artifacts/metrics.csv` using
the stdlib `csv` module. It never fabricates metric values, never asks the
LLM to estimate them, and never reads from a path that was not written by
the backtest engine during the current run.

## 9. `web_search`, `read_url` and `browser` are forbidden

For any prompt the dispatcher accepts, the only external tool invoked is
`backtest`. On success, failure, or exception, the dispatcher never calls
`web_search`, `read_url`, or any browser primitive.

## 10. LLM fallback is forbidden

If `write_config`, `write_signal_engine`, `backtest`, or `read_metrics`
fails, the dispatcher returns `{"status": "failed"}` with a sanitised
message. It never falls through to the LLM/ReAct loop as a recovery path.

## 11. Raw tracebacks are forbidden in any error path

The dispatcher calls `_sanitize()` on all exception messages before
returning them. `_sanitize()` applies a credential-pattern blocklist that
strips auth headers, bearer tokens, API keys, credential-prefixed strings,
Python tracebacks, internal URL schemes, absolute local filesystem paths,
and proxy addresses. Raw tracebacks or internal paths must never appear in
`result["content"]`.

## 12. Invented metrics are a regression

The dispatcher may only return `sharpe`, `max_drawdown`, `total_return`, and
`trade_count` values that were read from `artifacts/metrics.csv` and passed
through `float()` validation. Returning hard-coded, estimated, or LLM-generated
numbers is a regression equivalent to fabricating a verdict.

## Anti-regression test

The canary `agent/tests/test_baseline_backtest_workflow.py` has five
classes:

- **Class A — Document**: this file exists, contains the thirteen invariant
  substrings, does not leak credentials, and is bounded by identical SEALED
  markers.
- **Class B — Surface**: `backtest_intent`, `backtest_workflow`, and
  `BacktestWorkflowDispatcher` are present in code; `AgentLoop` integrates
  the dispatcher; `_MA_CROSSOVER_TEMPLATE` is referenced (source-level
  grep guard).
- **Class C — Workflow**: canonical prompt drives the dispatcher; trace
  records the five steps in exact order; metrics are read from `metrics.csv`
  and validated as numeric; no LLM is called.
- **Class D — Out-of-scope**: market data, candlestick, swarm, generic
  strategy, and ambiguous backtest prompts return `None` from `try_route` —
  the ReAct loop runs as before.
- **Class E — Fail-closed**: poisoned backtest error is sanitised against the
  credential-pattern blocklist; missing `metrics.csv` returns a closed failure;
  no web tool is invoked on any error path.

